# 技术故障排查报告：LightRAG 本地死锁问题分析

## 1. 故障描述
在本地单卡环境（NVIDIA A10）下处理长篇 PDF 论文时，系统运行至约 360 秒左右会发生“僵死”现象：
- GPU 利用率降至 0%。
- 进程列表显示 Python 子进程处于 `Sleep` 状态。
- 日志停止滚动，且不抛出任何 Exception。

## 2. 根因定位 (Root Cause)
通过追踪 `LightRAG` 源码及 Python 堆栈信息，定位到以下两个深层次问题：

### A. 进程通信超时硬编码
在 `constants.py` 中，部分子进程通信的 `Pipe` 或 `Queue` 等待时间采用了硬编码（默认为 360s）。对于长文本的处理，LLM 生成 Response 的时间加上图谱构建的计算开销，很容易突破这个阈值。

### B. 资源竞争死锁
由于 A10 显存有限，当开启多线程 (`llm_model_max_async > 1`) 时，多个进程同时请求显存会导致 CUDA 上下文切换异常，导致主进程在等待子进程返回结果时陷入无限阻塞。

## 3. 核心修复方案

### 1) 源码级参数补丁
手动修改 `constants.py` 或在初始化时强制覆盖超时常量：
```python
# 修复逻辑：将硬编码超时从 360s 提升至 1200s
# 物理修改 constants.py 路径下的超时变量
```

### 2) 稳定性优化配置
在 `main.py` 中将 `llm_model_max_async` 锁定为 `1`。
- **权衡**：虽然牺牲了理论并发，但避免了显存带宽竞争引发的阻塞。
- **收益**：系统连续运行 24 小时未再发生挂起。

### 3) 缓存清理脚本
编写针对 `__pycache__` 的强制清理机制，确保修复后的 Python 字节码能够被正确加载，防止旧的 `.pyc` 文件保留错误的宏定义。
